# Persisted Usage Analytics Plan

## Goal

Today `MonitoringController` (`GET /api/monitoring/presence`) exposes only **live, in-memory** WebSocket
presence — who is connected *right now*, with no history. We cannot tell whether a registered user is
"dead" or periodically active, nor when usage peaks during the day.

This adds **persisted** usage tracking, distinct from and additive to the existing live presence view:

1. **`users.last_seen_at`** — a timestamp bumped every time a user successfully opens a WebSocket
   connection. The dashboard opens `/ws` automatically on page load, so this is a reliable "user was
   active" signal. Directly answers *"is this specific user dead or alive?"*
2. **`connection_events`** — a new append-only table, one row per successful WebSocket connect
   (`user_id` + full raw `connected_at`). Enables historical queries (distinct users per day/hour,
   time-of-day patterns) via plain SQL.
3. **`GET /api/monitoring/usage`** — a new admin-only endpoint that aggregates `connection_events`
   into **time-slices** (default 30 min) over a date range, returning distinct-user counts per slice
   plus the most-active slice.

Nothing about the existing live `/api/monitoring/presence` endpoint changes — it stays in-memory only.

## Locked decisions (from requirements + clarification round)

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Store the **full raw `connected_at`** per connect. No bucketing, no dedup at write time. 10 reconnects in an hour = 10 rows. | Intentional write-side simplicity; all aggregation is read-time. |
| 2 | **No retention/pruning job.** Manual cleanup later if the table grows. | Keep scope minimal. |
| 3 | Both writes happen at **WebSocket connection open** (existing `@OnOpen`), not per message. | Low-frequency; ordinary JPA, far from the hot path (`binance/`, `feed/`, `analysis/`). |
| 4 | New schema via a **new Flyway migration** (`V15`). Never edit an applied migration. | Project convention; `baseline-version` is 4, latest is `V14`. |
| 5 | Persisted data is **distinct from** the live `/api/monitoring/presence` endpoint; expose via a **new endpoint** under `MonitoringController` (admin-only via existing `/api/monitoring/**` gate). | Requirement. |
| 6 | The usage endpoint returns **connection counts per time-slice**, not raw rows. Divide the range into slices (default 30 min, size configurable per request as an ISO-8601 duration). **Deduplicate per user within a slice** (distinct users). **Omit empty slices.** Include a field naming the **most-active slice**. | Readability requirement. |
| 7 | **Storage is always UTC.** `connected_at` / `last_seen_at` are `TIMESTAMPTZ` written from Java `Instant` (UTC), exactly like every other timestamp in the app. No local wall-clock is ever stored. | Correctness — one canonical timeline truth. |
| 8 | The report **`zone` is a query-time lens only.** It decides where slice boundaries fall (local midnight, local :00/:30) and labels the output. Optional request param, **defaults to `Asia/Tashkent`** (app's default region is UZ/UZS). | Time-of-day patterns must read in the users' local clock; changing `zone` re-slices the same stored data with no migration. |
| 9 | Request is **`GET` with query params** (consistent with existing `/presence`, `/orderbook`). | Pure read; matches the controller. |
| 10 | `start` and `end` are **inclusive calendar dates** interpreted in `zone`. Window = `[start 00:00 zone, (end+1 day) 00:00 zone)`. Same date on both = that one full local day. | Matches the admin's mental model ("July 18 only" → `start=end=2026-07-18`). |
| 13 | `start` and `end` are **optional**; when omitted they each default to **today in `zone`** (`LocalDate.now(zone)`). Omitting both → the current local day. | Convenience: hitting `/usage` bare gives today's usage. |
| 14 | Response includes **`totalConnections`** = the **sum of the per-slice `uniqueConnections`** across all returned slices. | Quick "how much usage in this range" scalar. |
| 11 | Record **only on a fully successful, entitled open** (after JWT *and* entitlement pass). A paywalled/expired user who loads the dashboard is **not** recorded. | Chosen in clarification: count active screener usage, not blocked attempts. |
| 12 | Also surface `last_seen_at` in the existing admin user listing (`GET /api/admin/users`). | Otherwise the column is write-only and can't answer "is *this* user dead." The user directory is the natural per-user read point. |

## Schema — `V15__add_last_seen_and_connection_events.sql`

```sql
ALTER TABLE users ADD COLUMN last_seen_at TIMESTAMPTZ;   -- nullable: NULL = never seen under new tracking

CREATE TABLE connection_events (
    id           BIGINT      GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id      UUID        NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    connected_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_connection_events_connected_at ON connection_events(connected_at);
```

- **`id` is `BIGINT ... IDENTITY`** (not UUID): this is a high-volume append-only log; a compact
  auto-increment PK is the natural fit. (Other tables use UUID PKs, but they are low-volume entities.)
- **`ON DELETE CASCADE`** mirrors `refresh_tokens` / `email_verification_tokens` — deleting a user
  removes their events, preserving referential integrity.
- **Index on `connected_at`** — the aggregation query is a range scan over this column.
- `last_seen_at` gets no DB default (consistent with the baselined `users` table, where the app sets
  timestamp values on write); existing rows stay `NULL` until first seen.

## New/changed Java — package `monitoring/` (+ small `user/` touches)

### New: `monitoring/ConnectionEvent.java` (entity)
Minimal JPA entity: `Long id` (`GenerationType.IDENTITY`), `UUID userId` (plain column, **not** a
`@ManyToOne` — no need to load `User`), `Instant connectedAt`. `@Getter`, `@NoArgsConstructor`, plus a
`(userId, connectedAt)` constructor.

### New: `monitoring/ConnectionEventRepository.java`
`JpaRepository<ConnectionEvent, Long>` with one native aggregation query:

```sql
SELECT FLOOR((EXTRACT(EPOCH FROM connected_at) - :windowStartEpoch) / :sliceSeconds)::bigint
           AS bucket_index,
       COUNT(DISTINCT user_id) AS unique_connections
FROM connection_events
WHERE connected_at >= :windowStart AND connected_at < :windowEnd
GROUP BY bucket_index
ORDER BY bucket_index
```

- Returns `List<Object[]>` — `[bucketIndex (bigint), uniqueConnections (bigint)]`.
- **Bucket alignment:** `bucket_index = floor((eventEpochSeconds − windowStartEpochSeconds) / sliceSeconds)`,
  where `windowStartEpoch = windowStart.getEpochSecond()` (computed in Java). Anchoring to
  `windowStart` (= local midnight of the start date) makes slices land on **local** :00/:30/day
  boundaries in any `zone`, not on the Unix epoch. Only non-empty buckets come back, already ordered.
- **Why `EXTRACT(EPOCH FROM connected_at) - :windowStartEpoch` (numeric) instead of subtracting a
  timestamp param:** `EXTRACT(EPOCH FROM timestamptz)` is unambiguous Unix seconds; subtracting a
  plain `long` avoids any `timestamptz`/`timestamp`-without-zone binding ambiguity for the anchor. The
  only `Instant` params are the `WHERE` range bounds (the standard, safe pattern).

### New: `monitoring/ConnectionActivityService.java` (write path)
```java
@Service @RequiredArgsConstructor
class ConnectionActivityService {
    @Transactional
    void recordConnection(UUID userId) {
        Instant now = Instant.now();
        userRepository.updateLastSeenAt(userId, now);      // bulk UPDATE, no entity load
        connectionEventRepository.save(new ConnectionEvent(userId, now));
    }
}
```
Both writes share one transaction. Same `Instant` used for both so `last_seen_at` equals the last
event's `connected_at`.

### New: `monitoring/ConnectionUsageService.java` (read/report path)
`@Transactional(readOnly = true)`. `report(LocalDate start, LocalDate end, String sliceIso, String zoneId)`
— `start`/`end` may be `null`:
1. Parse `zoneId` → `ZoneId` (bad value → `ApiException.badRequest`).
2. Parse `sliceIso` → `Duration.parse` (bad value → `badRequest`); require `>= PT1M` (guards against
   divide-by-zero and absurdly fine slices).
3. **Default dates:** `if (start == null) start = LocalDate.now(zone); if (end == null) end = LocalDate.now(zone);`
   (defaulting is done *after* the zone is resolved, so "today" means today in `zone`).
4. Validate `end >= start` (else `badRequest`).
4. `windowStart = start.atStartOfDay(zone).toInstant()`;
   `windowEnd = end.plusDays(1).atStartOfDay(zone).toInstant()` (**end date inclusive**).
5. `sliceSeconds = slice.getSeconds()`, `windowStartEpoch = windowStart.getEpochSecond()`.
6. Call the repository; map each row to a `UsageSlice` whose start/end are
   `windowStart.plusSeconds(index * sliceSeconds)` projected into `zone` as `OffsetDateTime`
   (human-readable, e.g. `2026-07-18T15:00+05:00`). Accumulate `totalConnections += uniqueConnections`.
7. Track the **most-active** slice (max `uniqueConnections`; ties → earliest, since rows are ordered).
8. Build the response.

> `totalConnections` is the **sum of the per-slice distinct counts**, *not* distinct users across the
> whole range — a user active in two slices contributes to both. (Distinct-users-over-range would be a
> separate query; not requested.)

> Note: `Duration.parse` accepts `PT30M`, `PT1H`, `P1D`, `PT168H`, `P7D`, etc. It does **not** accept
> `P1W` (weeks) or months/years — those aren't fixed-length durations anyway. For a week the admin
> sends `P7D`. This will be documented in the endpoint contract.

### New: `monitoring/dto/UsageReportResponse.java`
```java
public record UsageReportResponse(
        OffsetDateTime start,        // window start, in zone
        OffsetDateTime end,          // window end (exclusive), in zone
        String zone,                 // e.g. "Asia/Tashkent"
        String slice,                // echoed ISO-8601 duration, e.g. "PT30M"
        int sliceCount,              // number of non-empty slices returned
        long totalConnections,       // sum of uniqueConnections across all returned slices
        UsageSlice mostActive,       // null when the range has no connections
        List<UsageSlice> slices) {

    public record UsageSlice(OffsetDateTime start, OffsetDateTime end, long uniqueConnections) {}
}
```
(DTOs go in a `dto/` subpackage per project convention.)

### Changed: `monitoring/MonitoringController.java`
Add `ConnectionUsageService` to the constructor and a new endpoint (existing endpoints untouched):
```java
@GetMapping("/usage")
public UsageReportResponse getUsage(
        @RequestParam(required = false) @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) LocalDate start,
        @RequestParam(required = false) @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) LocalDate end,
        @RequestParam(defaultValue = "PT30M") String slice,
        @RequestParam(defaultValue = "Asia/Tashkent") String zone) {
    return connectionUsageService.report(start, end, slice, zone);  // null start/end default to today in zone
}
```
Already admin-only via the `/api/monitoring/**` matcher in `SecurityConfig` — no security change needed.
Also update the class Javadoc (which currently says persisted metrics are "future work").

### Changed: `ws/ScreenerWebSocketEndpoint.java`
Inject `ConnectionActivityService` (via `@Autowired`, matching the other fields — this endpoint is a
Spring-managed `@ServerEndpoint`). In `onOpen`, **after** the session is fully established
(`broadcaster.addSession(userSession)`), i.e. strictly after the entitlement check passed:
```java
try {
    connectionActivityService.recordConnection(user.userId());
} catch (Exception e) {
    log.warn("Failed to record connection activity for user {}: {}", user.userId(), e.getMessage());
}
```
Wrapped in try/catch so an analytics DB hiccup never tears down a live user session.

### Changed: `user/UserRepository.java`
```java
@Modifying
@Query("update User u set u.lastSeenAt = :ts where u.id = :id")
void updateLastSeenAt(@Param("id") UUID id, @Param("ts") Instant ts);
```
Bulk update — avoids loading the `User` entity on the connect path.

### Changed: `user/User.java`
Add `@Column(name = "last_seen_at") private Instant lastSeenAt;` (nullable; `@Getter/@Setter` cover it).
No `@PrePersist` change — a brand-new user has no last-seen until they connect.

### Changed: `user/dto/AdminUserView.java` + `user/UserAdminService.java`
Add `Instant lastSeenAt` as the final record field; pass `u.getLastSeenAt()` in `toView(...)`. This is
the only per-user read surface for the new column. (Frontend contract note: the admin user row gains
one nullable field.)

## Request / response contract

**`GET /api/monitoring/usage`** — admin only.

| Param | Required | Default | Notes |
|-------|----------|---------|-------|
| `start` | no | today in `zone` | ISO date `YYYY-MM-DD`, interpreted in `zone`. Inclusive. |
| `end`   | no | today in `zone` | ISO date. Inclusive (window extends through end of that local day). |
| `slice` | no | `PT30M` | ISO-8601 duration, `>= PT1M`. E.g. `PT30M`, `PT1H`, `P1D`, `P7D`. |
| `zone`  | no | `Asia/Tashkent` | IANA zone id. Used only to place slice boundaries + label output. |

**One-day example:** `GET /api/monitoring/usage?start=2026-07-18&end=2026-07-18` (equivalently, a bare
`GET /api/monitoring/usage` on 2026-07-18 in `Asia/Tashkent`) →
```json
{
  "start": "2026-07-18T00:00+05:00",
  "end":   "2026-07-19T00:00+05:00",
  "zone":  "Asia/Tashkent",
  "slice": "PT30M",
  "sliceCount": 2,
  "totalConnections": 8,
  "mostActive": { "start": "2026-07-18T18:00+05:00", "end": "2026-07-18T18:30+05:00", "uniqueConnections": 5 },
  "slices": [
    { "start": "2026-07-18T15:00+05:00", "end": "2026-07-18T15:30+05:00", "uniqueConnections": 3 },
    { "start": "2026-07-18T18:00+05:00", "end": "2026-07-18T18:30+05:00", "uniqueConnections": 5 }
  ]
}
```
Empty range → `totalConnections: 0`, `mostActive: null`, `slices: []`, `sliceCount: 0`.

**Error cases** (`ApiException` → `ApiError` shape): `end < start`, unparseable `slice`, `slice < PT1M`,
invalid `zone` → `400 Bad Request`.

## What is deliberately NOT built

- No retention/pruning job (decision 2).
- No write-time dedup or bucketing (decision 1).
- No change to the live `/api/monitoring/presence` endpoint.
- No `P1W`/month/year slice support (not fixed-length; use `P7D` for a week).
- No new config knob — defaults (`PT30M`, `Asia/Tashkent`) are endpoint literals; can be promoted to
  `screener.*` config later if needed.
